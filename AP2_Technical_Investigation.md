## 2. プロトコル概要

### 2.1 ACP vs AP2 比較

| 項目 | ACP | AP2 |
|------|-----|-----|
| **正式名称** | Agentic Commerce Protocol | Agent Payments Protocol |
| **主導者** | OpenAI / Stripe | Google / Mastercard / Visa / JCB |
| **本格展開** | 2025年〜（ChatGPTで稼働中） | 2026年以降 |
| **決済モデル** | 即時決済（同期） | 自律購買（非同期対応） |
| **主なユースケース** | B2C、少額決済、チャット経由購入 | B2B、高額取引、非同期決済 |
| **認証方式** | HMAC-SHA256署名検証 | 暗号署名による信頼基盤 |

## 3. MCP（Model Context Protocol）との関係

### 3.1 MCPとは

MCPはAnthropicが推進するオープンプロトコルで、AIアプリケーションと外部システムを接続する標準規格です。

**主な特徴：**
- オープンソース（Linux Foundation管理）
- 多言語SDK対応（TypeScript, Python, Java, Kotlin, C#, Go, PHP, Ruby, Rust, Swift）
- Claude, ChatGPT, VS Code, Cursorなど主要AIツールがサポート

**公式リソース：**
- ドキュメント: https://modelcontextprotocol.io
- GitHub: https://github.com/modelcontextprotocol

### 3.2 MCPにおける決済の位置づけ

| 項目 | 内容 |
|------|------|
| **コア機能** | 決済専用機能はコアに含まれず |
| **拡張方法** | カスタムMCPサーバーとして実装 |
| **サードパーティ** | PayMCP等のフレームワークが存在 |
| **セキュリティ** | Authorization Extensions, Security Best Practicesを活用 |

### 3.3 PayMCP

```
PayMCP: MCPサーバーに決済機能を追加する軽量フレームワーク
- Python & TypeScript対応
- 2行のデコレーターでツールを有料エンドポイント化
```

## 4. AP2実装に向けた技術要件

### 4.1 推定される技術スタック

```yaml
AP2 PSP Gateway 構成要素:

  認証・認可:
    - 暗号署名（公開鍵/秘密鍵ペア）
    - OAuth 2.0 / OpenID Connect
    - JWTトークン

  API:
    - RESTful API または gRPC
    - Webhook通知
    - 冪等性キー対応

  セキュリティ:
    - TLS 1.3
    - PCI DSS準拠
    - 署名検証

  非同期対応:
    - イベント駆動アーキテクチャ
    - メッセージキュー（Kafka/RabbitMQ等）
    - 状態管理
```

### 4.2 ACP実装仕様（参考）

ACPの実装仕様を参考に、AP2でも同様の要件が求められると推測：

| 実装項目 | ACP仕様 | AP2推測 |
|----------|---------|---------|
| エンドポイント | `POST /agentic_commerce/delegate_payment` | 類似のRESTful API |
| 通信要件 | TLS1.2以上 | TLS1.3推奨 |
| 署名検証 | HMAC-SHA256 | 公開鍵暗号（RSA/ECDSA） |
| 冪等性 | Idempotency-Key対応 | 同様 |
| トークン | 一回限りの制約付きトークン | より高度な暗号トークン |

## 5. 日本独自決済との親和性分析

### 5.1 非同期決済対応

AP2は非同期決済との高い親和性を持つ：

| 決済手段 | ACP適合性 | AP2適合性 | 理由 |
|----------|-----------|-----------|------|
| クレジットカード | ◎ | ○ | 即時決済、両方対応 |
| コンビニ決済 | △ | ◎ | 非同期、AP2向き |
| 銀行振込 | △ | ◎ | 非同期、AP2向き |
| 後払い決済 | △ | ◎ | 非同期、AP2向き |
| キャリア決済 | ○ | ○ | 両方対応可 |

### 5.2 技術的な親和性ポイント

```
AP2の非同期決済サポート

1. 状態管理
   ├── PENDING: 支払い待ち
   ├── PROCESSING: 処理中
   ├── COMPLETED: 完了
   └── FAILED: 失敗

2. イベント通知
   ├── Webhook: 状態変更時にPSPからAIエージェントへ通知
   └── ポーリング: エージェントからの状態確認

3. タイムアウト処理
   └── コンビニ決済（〜72時間）などの長期待機に対応
```

### 5.3 SPLinkの競争優位性

| 要素 | 詳細 |
|------|------|
| **日本特有決済の網羅** | コンビニ、銀行振込、JCB、キャリア決済等 |
| **非同期処理の実績** | 既存システムで非同期決済を運用中 |
| **加盟店ネットワーク** | 数万店舗との既存連携 |
| **商習慣の理解** | 日本市場特有の要件への対応力 |

## 8. 技術調査で判明した課題

### 8.1 情報の制約

| 課題 | 詳細 | 対応策 |
|------|------|--------|
| AP2仕様未公開 | 詳細な技術仕様は非公開 | パートナー登録で情報取得 |

### 8.2 技術的な検討事項

```
要検討項目:

1. 暗号署名方式
   └── RSA vs ECDSA vs Ed25519

2. 非同期処理アーキテクチャ
   └── イベント駆動 vs ポーリング

3. 冪等性の実装
   └── サーバーサイドのキー管理

4. エラーハンドリング
   └── 決済失敗時のリトライ戦略
```
